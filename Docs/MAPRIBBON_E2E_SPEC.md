## 0. 문서 목적

이 문서는 여행 중 촬영한 사진의 촬영 시각과 위치 메타데이터를 이용해, 사용자의 하루를 지도·사진·핀·연결선으로 자동 구성하는 iOS 앱의 제품 방향, 기능 범위, UI 설계, 기술 구조, 데이터 모델, 권한 정책, 수익화, 한국·일본 출시 전략까지 정의한다.

이 앱은 GPS 추적기, 여행 일정 관리 앱, 여행 SNS가 아니다. 핵심 결과물은 다음 한 문장으로 정의한다.

> **하루 동안 찍은 사진을 골라볼 필요 없이, 날짜만 누르면 한 장의 여행 포토보드가 완성된다.**

# 1. 제품 요약

## 1.1 제품 정의

MapRibbon은 iPhone 사진 보관함에서 위치가 포함된 사진을 날짜별로 분석하고, 가까운 촬영 지점을 하나의 방문 장소로 묶은 뒤, 시간순으로 연결한 **하루 여행 포토보드**를 자동 생성한다.

사용자는 여행 중 앱을 켜거나 기록을 시작하지 않는다. 여행이 끝난 뒤 앱을 열어 날짜를 선택하고, 자동 생성된 결과를 확인한 다음 바로 공유한다.

### 핵심 흐름

```text
앱 실행
→ 위치 사진이 있는 날짜 확인
→ 날짜 선택
→ 장소별 자동 묶기
→ 대표 사진 자동 선택
→ 지도·핀·리본 연결선으로 보드 생성
→ 필요 시 사진/장소명만 수정
→ 이미지 공유
```

## 1.2 제품이 해결하는 문제

여행 사진은 많이 남지만 다음 문제가 반복된다.

- 사진 보관함 안에서 날짜와 장소 맥락이 사라진다.
- 여행을 기록하려면 별도 일기를 작성하거나 사진을 수동 배치해야 한다.
- 지도 앱은 장소 탐색에 강하지만, 공유 가능한 결과물을 만들지 않는다.
- 여행 기록 앱은 여행 생성, 로그인, GPS 추적, 글 작성 등 시작 마찰이 크다.
- 기본 사진 앱의 추억 기능은 결과 형식과 사진 구성을 사용자가 통제하기 어렵다.

MapRibbon은 기록 행위를 추가하지 않고, 이미 촬영한 사진을 결과물로 바꾼다.

## 1.3 제품이 추구하는 방향

### 방향 A. 기록 도구보다 결과물 생성기

앱 안에서 오래 머무르게 하지 않는다. 사용자가 원하는 것은 여행 데이터를 관리하는 일이 아니라, 보기 좋게 정리된 한 장이다.

### 방향 B. 정확한 이동 경로보다 기억의 순서

핀 사이의 리본 선은 실제 도로나 GPS 궤적을 의미하지 않는다. 사진 촬영 장소의 시간순 관계를 보여주는 시각적 장치다.

### 방향 C. 자동 생성 우선, 편집은 예외 처리

사용자가 Canva처럼 직접 배치하는 앱이 아니다. 기본 결과가 충분히 좋아야 하며, 편집은 대표 사진 교체·장소명 수정·묶음 합치기 정도로 제한한다.

### 방향 D. 가입·광고·추적 없는 로컬 앱

사진과 위치는 기기 안에서 처리한다. 계정, 서버 업로드, 광고 SDK, 백그라운드 위치 추적을 초기 버전에서 사용하지 않는다.

### 방향 E. 하루 결과와 누적 지도를 함께 제공

단기 사용 이유는 하루 포토보드이고, 장기 재방문 이유는 누적되는 **Memory Atlas**다. 한국에서는 17개 시·도, 일본에서는 47개 도도부현을 사진으로 채우는 누적 지도를 제공한다.

# 2. 벤치마크: 旅行思い出マップ

## 2.1 확인된 제품 구조

일본 앱 `旅行思い出マップ - 旅の記憶を、地図に描く`은 사진을 행정구역 형태로 잘라 지도에 채우는 방식으로 여행 기록을 작품화한다. 일본·아시아·도쿄·홋카이도 등 여러 지도 단위를 제공하며, 리스트 보기, 복수 사진, 댓글, 공동 편집 링크, 스티커, 이미지 공유 기능을 확장해 왔다.

2026년 7월 조사 시점의 앱스토어 설명에는 약 11.7만 다운로드, 4.6점대 평가, 일본 47개 도도부현과 세부 지역 지도, 공동 편집과 댓글 기능이 표시되어 있다. 최근 업데이트에서는 커뮤니티 탭을 제거하고 지도 제작에 다시 집중했으며, 스티커·지역 지도·구독 관리 기능을 추가했다.

## 2.2 인기 요소 분석

### 1. 빈 지도를 채우는 수집 감각

사진 한 장을 추가할 때마다 지도에서 비어 있던 지역이 채워진다. 사용 완료가 아니라 완성도를 높이는 행동이므로, 사용자가 다음 여행을 떠날 이유까지 만든다.

### 2. 사진과 장소가 한눈에 결합

일반적인 여행 기록은 사진과 지도 사이를 전환해야 한다. 이 앱은 사진 자체를 지도의 일부로 만들어 시각적으로 즉시 이해된다.

### 3. 결과물이 개인 작품처럼 보임

여행 데이터가 아니라 테마별 작품으로 인식된다. 지역 음식, 야경, 기념품처럼 동일 지역도 여러 테마로 다시 만들 수 있다.

### 4. 지도 보기와 리스트 보기의 병행

지도에서는 전체 진행 상태를 보고, 리스트에서는 각 지역의 사진을 명확하게 확인한다. 공간적 보기와 관리용 보기를 분리한 구조가 유효하다.

### 5. 지역 확장이 콘텐츠 업데이트가 됨

도쿄 62개 시구정촌, 홋카이도 179개 시정촌 같은 세부 지도를 추가하는 것만으로도 앱 업데이트와 재방문 이유가 생긴다.

## 2.3 벤치마크에서 가져올 것

| 요소 | MapRibbon 적용 방식 |
|---|---|
| 사진으로 지도를 채우는 성취감 | Memory Atlas에서 방문 지역을 대표 사진으로 자동 채움 |
| 지도/리스트 이중 보기 | 보드 보기와 방문 장소 목록 보기 제공 |
| 테마별 작품 | 기본 템플릿과 시즌·도시별 보드 스타일로 전환 |
| 지역 세분화 | 한국 17개 시·도 → 제주/서울 세부 지도, 일본 47개 도도부현 순서로 확장 |
| 이미지 공유 | 스토리·피드·일반 포스터 비율 지원 |
| 스티커 | 과도하지 않은 우표·날짜 스탬프·티켓 조각으로 제한 |
| 누적 완성도 | 방문 지역 수, 생성한 하루 보드 수, 사진 장소 수 표시 |

## 2.4 가져오지 않을 것

| 제외 항목 | 이유 |
|---|---|
| 가입 필수 | 첫 결과 생성 전 마찰 증가 |
| 광고 시청 후 기능 사용 | 사진 접근 권한 직후 광고가 나오면 신뢰와 전환이 급격히 떨어짐 |
| 공개 커뮤니티 | 신고, 차단, 콘텐츠 검수, 서버 운영 필요 |
| 댓글·채팅 | 핵심 결과물 생성과 무관하며 계정·동기화가 필요 |
| 실시간 공동 편집 | 서버 또는 CloudKit 공유 설계가 필요하고 초기 구매 이유와 거리가 있음 |
| 사진 한 장씩 행정구역에 수동 배치 | 작업량이 많고 하루 자동 생성 컨셉과 충돌 |
| 월 구독 중심 | 사용 빈도가 여행 시점에 집중되므로 영구 구매가 더 자연스러움 |

## 2.5 MapRibbon의 차별점

`旅行思い出マップ`은 장기간에 걸쳐 지역을 채우는 컬렉션 앱이다. MapRibbon은 **하루의 사진을 시간순 여행 포스터로 즉시 변환**하는 앱이다.

| 구분 | 旅行思い出マップ | MapRibbon |
|---|---|---|
| 핵심 단위 | 국가·현·도시 지도 | 하루 |
| 입력 | 지역별 사진 수동 추가 | 날짜 선택 후 자동 분석 |
| 핵심 결과 | 방문 지역이 채워진 누적 지도 | 한 장의 하루 여행 포토보드 |
| 지도 의미 | 행정구역 컬렉션 | 촬영 장소의 시간순 관계 |
| 사용 시점 | 여행 후 지속적으로 채움 | 여행 당일 또는 다음 날 10초 내 생성 |
| 운영 구조 | 계정·광고·공동 편집 | 로컬·무가입·무광고 |
| 장기 유지 | 지역 완성률 | Memory Atlas + 일별 보드 아카이브 |

# 3. 앱 이름과 브랜드

## 3.1 최종 작업명: MapRibbon

### 의미

- **Map**: 사진이 촬영된 장소
- **Ribbon**: 시간순으로 이어지는 하루의 흐름
- 지도 핀을 실이나 리본으로 연결한 여행 보드의 핵심 시각 언어를 직접 표현한다.

### 발음

- 한국: 맵리본
- 일본: マップリボン
- 영어: Map Ribbon

### 스토어 노출명

#### 한국

**맵리본 - 사진으로 만드는 여행 지도**

부제 후보:

- 여행 사진 자동 지도·포스터
- 하루 여행을 한 장으로
- 사진 위치로 만드는 여행 기록

#### 일본

**MapRibbon - 写真でつくる旅ボード**

サブタイトル 후보:

- 1日の旅を一枚のフォトマップに
- 写真の場所と時間から自動作成
- 登録不要の旅ポスター作成

## 3.2 대체 후보

| 이름 | 장점 | 약점 |
|---|---|---|
| DayRibbon | 하루 단위가 분명함 | 지도 앱인지 즉시 알기 어려움 |
| TripCanvas | 결과물이 작품이라는 의미 | 유사 서비스명 가능성이 높음 |
| MapMemoir | 기억과 지도의 결합 | 한국·일본 발음이 길음 |
| Journey Thread | 핀과 실 연결을 잘 표현 | 앱 이름으로 길고 검색성이 약함 |
| 하루여정 | 한국에서는 즉시 이해 | 일본·영어 확장 시 별도 브랜드 필요 |

## 3.3 네이밍 주의

MapRibbon은 현재 작업명이다. 출시 전 다음 검증이 필요하다.

- App Store Connect 이름 예약 가능 여부
- 한국·일본 상표 검색
- 주요 도메인과 SNS 핸들 확보 가능 여부
- 일본어 표기에서 부정적·중의적 의미가 없는지 확인

# 4. 목표 사용자와 사용 상황

## 4.1 1차 타깃: 한국

### 핵심 사용자

- 해외여행·국내 당일치기 후 사진을 인스타그램 스토리나 카카오톡으로 공유하는 20~40대
- 여행 일기는 쓰지 않지만 사진을 보기 좋게 남기고 싶은 사용자
- 카페·전시·문구점·맛집 투어를 하루 단위로 다니는 사용자
- 아이폰 사진 위치정보를 켜두는 사용자
- 복잡한 편집 앱을 사용하지 않는 사용자

### 대표 사용 상황

1. 도쿄 여행 2일 차 사진으로 스토리용 지도 만들기
2. 서울 전시회·카페·식당을 다닌 토요일 기록
3. 제주 동쪽 하루 코스를 가족 단톡방에 공유
4. 문구점 투어 결과를 SNS 피드 포스터로 정리
5. 한 달 뒤 여행 사진을 다시 찾다가 자동 보드 생성

## 4.2 2차 타깃: 일본

### 핵심 사용자

- 国内旅行, 日帰り旅行, 街歩き, カフェ巡り를 기록하는 사용자
- 旅日記를 쓰고 싶지만 입력 부담이 큰 사용자
- 사진 콜라주와 지도형 기록을 좋아하는 사용자
- 도도부현 수집, 방문 지도 완성에 반응하는 사용자
- 계정 생성과 구독을 피하는 사용자

### 일본 현지 사용 문구

- 写真を選ばなくても、旅の一日が一枚になる
- 位置情報付き写真から、自動で旅ボードを作成
- 登録不要・広告なし・端末内で完結
- 47都道府県の思い出マップも少しずつ完成

# 5. 핵심 제품 강점

## 5.1 사전 동작이 없음

- 여행 생성 불필요
- 기록 시작 불필요
- 백그라운드 위치 추적 없음
- 앱 카메라 사용 불필요
- 일정 입력 불필요

## 5.2 날짜 한 번 선택으로 자동 완성

사진 권한을 허용하면 위치 사진이 있는 날짜를 자동 탐색한다. 사용자는 사진 수십 장을 직접 고르지 않는다.

## 5.3 실제 경로를 주장하지 않는 시각 언어

핀과 리본 선은 시간 순서를 뜻한다. 실제 도로를 따라가지 않기 때문에 GPS 궤적 정확도, 배터리, 교통수단 추정 문제가 사라진다.

## 5.4 결과물이 앱의 광고가 됨

공유 이미지 하단에 강한 워터마크를 넣지 않는다. 무료 버전에서는 작은 `Made with MapRibbon` 서명만 사용하고 유료에서 제거한다.

## 5.5 장기적으로 채워지는 Memory Atlas

하루 보드를 만들 때마다 방문한 지역이 누적 지도에 자동 반영된다. 별도 입력 없이 장기 사용 동기가 생긴다.

## 5.6 로컬 처리와 낮은 운영비

- 서버 없음
- 계정 없음
- 사진 업로드 없음
- 광고 SDK 없음
- 외부 AI API 없음
- 지도 스냅샷 외 대부분 오프라인 처리

# 8. 핵심 사용자 플로우

```text
인트로 1장

→ 필요 시 수정
→ 공유
→ 공유 완료 후 Memory Atlas 갱신 안내
```

### 원칙

- 3장 이상의 온보딩 금지
- 권한 요청 전에 결과 예시와 사용 이유를 보여줌

- 생성 전 템플릿 선택 없음
- 첫 결과 후 구매 화면을 강제로 띄우지 않음

```text
앱 실행
→ “어제 위치 사진 34장, 장소 5곳” 카드
→ 만들기
→ 자동 생성
→ 공유
```

목표는 앱 실행 후 결과 화면까지 평균 10초 이내다. iCloud 원본 다운로드가 필요한 경우 메타데이터와 저해상도 썸네일로 먼저 미리보기를 만든다.

## 8.3 사진 권한을 제한적으로 허용한 사용자

```text
선택한 사진으로 만들기
→ Photos Picker

```

전체 자동 날짜 탐색은 불가능하므로 다음 문구를 제공한다.

> 선택한 사진만으로 보드를 만들고 있습니다. 날짜별 자동 생성을 사용하려면 사진 접근 범위를 넓혀주세요.

# 9. UI/UX 디자인 명세

## 9.1 디자인 모드

**Productive Minimal Application**을 주 모드로 사용한다.

이 앱의 핵심은 편집 도구가 아니라 결과물 확인과 공유다. 화면은 한 가지 주 행동을 분명하게 보여주고, 보드 자체가 가장 강한 시각 요소가 되어야 한다.

## 9.2 디자인 테제

> 앱 UI는 조용하고 기능적이며, 생성된 여행 보드는 인쇄물처럼 보인다. 앱의 장식이 사진을 이기지 않고, 한 화면에서 사용자가 해야 할 행동은 하나만 남긴다.

## 9.3 시각 방향

### 앱 UI

- 밝은 중성 배경
- 얇은 경계선
- 8~12pt 반경
- 사진과 지도 미리보기에만 그림자
- 하단 주요 CTA 1개
- SF Symbols 기반 아이콘
- 과도한 그래디언트·글래스·입체 버튼 금지

### 출력 보드

- 지도 종이 또는 크림색 보드
- 사진 카드 4~8개
- 핀 또는 작은 원형 노드
- 리본 연결선
- 날짜 스탬프
- 도시명과 장소명
- 선택적으로 우표·티켓 조각

앱 UI와 출력 보드를 같은 스타일로 만들지 않는다. 앱은 도구, 보드는 결과물이다.

# 10. 화면별 설계

## S01. 첫 실행 인트로

### 목적

앱의 결과물을 3초 안에 이해시키고 사진 권한 요청으로 자연스럽게 연결한다.

### 구성

- 상단 작은 로고
- 중앙 샘플 여행 보드
- 헤드라인: `찍어둔 사진이, 하루의 여행 지도가 됩니다.`
- 보조 문구: `기록 시작도, 위치 추적도 필요 없습니다.`
- CTA: `내 사진으로 만들어보기`
- 하단 작은 문구: `사진과 위치는 기기 안에서 처리됩니다.`

### 수용 조건

- 첫 화면에서 실제 보드 결과가 가장 크게 보임
- 슬라이드 온보딩 없음
- 건너뛰기 버튼 없음

## S02. 사진 권한 설명

### 목적

시스템 권한 팝업 전에 사용 목적을 명확하게 전달한다.

### 구성

- 사진 3장이 지도 핀으로 변환되는 간단한 도식
- 제목: `사진의 날짜와 장소를 읽어 보드를 만듭니다.`
- 설명:
  - 원본은 업로드하지 않음
  - 백그라운드 위치 추적 없음
  - 제한 접근도 지원
- CTA: `사진 접근 계속`
- 보조 CTA: `사진 직접 선택`

## S03. 날짜 목록

### 목적

사용자가 보드로 만들기 좋은 하루를 빠르게 선택한다.

### 레이아웃

- 상단 제목 `어느 하루를 남길까요?`
- 추천 카드 1개
- 월별 날짜 리스트
- 각 행:
  - 날짜
  - 도시/지역 추정
  - 위치 사진 수
  - 장소 클러스터 수
  - 사진 3장 미니 프리뷰

### 추천 카드 예시

```text
어제 · 도쿄
사진 42장 · 장소 6곳
[보드 만들기]
```

### 필터

- 최근
- 여행 가능성이 높은 날
- 전체 날짜

`여행 가능성이 높은 날`은 서로 2km 이상 떨어진 장소가 3개 이상이고, 촬영 기간이 3시간 이상인 날짜로 정의한다.

## S04. 분석 중

### 목적

작업 상태를 짧고 투명하게 표시한다.

### 단계 문구

1. 사진의 시간과 장소 확인
2. 가까운 사진 묶기
3. 대표 사진 고르기
4. 보드 구성

진행률 숫자는 실제 단계 기반으로 계산한다. 가짜 99% 진행률 금지.

## S05. 보드 미리보기

### 목적

결과 확인과 공유를 최우선으로 제공한다.

### 계층

1. 보드 전체 미리보기
2. `공유` CTA
3. 수정·템플릿 변경

### 레이아웃

- 상단: 뒤로, 제목, 더보기
- 중앙: 세로 보드 미리보기
- 하단 고정 CTA: `공유하기`
- 보조 버튼: `사진/장소 수정`, `스타일`

### 제스처

- 미리보기 탭: 전체 화면
- 핀 탭: 해당 장소 사진 시트
- 사진 길게 누름: 대표 사진 교체

- 장소명 수정
- 대표 사진 교체
- 장소 묶음 합치기
- 사진을 다른 장소로 이동
- 위치 없는 사진 추가
- 장소 삭제

### 금지 기능

- 자유 회전
- 무제한 스케일
- 모든 요소 자유 배치
- 레이어 패널
- 텍스트 스타일 세부 편집

## S07. 템플릿 선택

### MVP 템플릿

1. **Ribbon Map**: 지도 중심, 사진 카드와 리본 선
2. **Editorial Day**: 큰 대표 사진 + 작은 지도 + 장소 타임라인
3. **Postcard**: 사진 중심, 최소 지도
4. **Scrap Map**: 종이 지도, 티켓·스탬프, 얇은 연결선

- 3곳 이하
- 4~5곳
- 6~7곳
- 8곳 이상

8곳 이상은 모두 표시하지 않고 중요도 순으로 7개까지 축약한다.

# 11. 자동 분석 로직

## 11.1 입력 데이터

각 `PHAsset`에서 다음을 사용한다.

- localIdentifier
- creationDate
- location
- mediaType
- mediaSubtypes
- pixelWidth / pixelHeight
- favorite
- hidden
- modificationDate

스크린샷, 화면 녹화, 밈·저장 이미지 가능성이 높은 자산은 기본 대표 후보에서 제외한다.

## 11.2 날짜 그룹화

### 기본 규칙

- 사용자의 현재 캘린더와 타임존 기준 날짜
- 여행지 현지 날짜 보정은 사진 좌표의 타임존을 역추정해 적용 가능
- MVP에서는 자정 전후 3시간 이내에 같은 도시에서 이어진 사진을 하나의 하루 후보로 합칠 수 있도록 사용자 선택 제공

## 11.3 장소 클러스터링

### 1차 공간 클러스터

- 기본 반경: 180m
- 도심 밀집 지역: 100m
- 교외: 300m

### 2차 시간 분리

같은 지역이라도 촬영 간격이 2시간 이상이고 중간에 다른 장소가 있으면 별도 방문으로 분리한다.

### 병합 조건

- 중심점 거리 120m 이하
- 시간 구간이 겹치거나 45분 이내
- 역지오코딩 POI 또는 행정 주소가 동일

### 출력

각 클러스터는 `VisitCluster`가 된다.

## 11.4 대표 사진 선정

### 제외 우선순위

- 스크린샷
- 중복·연사 유사 이미지
- 심하게 흐린 이미지
- 극단적 저노출
- QR·영수증·문서 중심 이미지

### 가점

- 즐겨찾기
- 사용자가 편집한 사진
- 얼굴 또는 주요 피사체가 선명함
- 장소를 설명할 수 있는 넓은 장면
- 템플릿 비율에 적합
- 같은 클러스터 내 다른 사진과 시각적 중복이 적음

### Vision 활용

- Feature Print로 유사 사진 묶기
- 얼굴 품질 점수는 인물 사진 후보에만 적용
- 텍스트 비중이 높은 사진은 OCR 결과를 이용해 대표 후보 점수 하향

Vision 분석은 썸네일 해상도로 수행하고, 생성 중 UI가 멈추지 않도록 actor 기반 백그라운드 작업으로 분리한다.

## 11.5 장소명 생성

우선순위:

1. 지도 POI명
2. 역·공원·박물관 등 대표 장소
3. 동/구/시 단위 주소
4. 사용자가 수정한 이름

개인 주거지 가능성이 있는 장소는 상세 주소 대신 동·구 수준으로 축약한다.

## 11.6 연결선

- 실제 경로 계산 없음
- 장소 중심점을 시간순으로 연결
- 직선 또는 1개 제어점을 가진 부드러운 곡선
- 선이 사진 카드를 관통하지 않도록 충돌 회피
- 이동 거리가 긴 구간은 점선
- 같은 건물 내 이동은 연결 생략 가능

## 11.7 자동 레이아웃

### 원칙

지도 좌표를 그대로 사진 카드 위치로 쓰지 않는다. 지도 핀은 실제 위치를 유지하되 사진 카드는 가독성을 위해 주변 안전 영역으로 이동하고 리더 라인으로 연결한다.

### 레이아웃 단계

1. 지도 스냅샷 영역 결정
2. 핀 좌표를 이미지 좌표로 변환
3. 사진 카드 후보 영역 계산
4. 카드 간 겹침 최소화
5. 제목·날짜·통계 영역 보호
6. 연결선 충돌 최소화
7. 결과 점수 평가
8. 사전 정의된 레이아웃 중 최고 점수 선택

완전한 물리 엔진 대신, 사진 수 구간별 6~8개 레이아웃 프리셋을 두고 핀 위치에 따라 가장 적합한 프리셋을 선택한다.

# 14. 권한과 개인정보

## 14.1 필요한 권한

### 사진 읽기

필수 자동 기능에 사용한다.

권한 설명 예시:

> 선택한 날짜의 사진과 촬영 위치를 읽어 여행 포토보드를 만듭니다. 사진은 서버로 전송되지 않습니다.

### 사진 추가

결과물을 사진 앱에 직접 저장할 때만 요청한다. 공유 시트만 사용하는 경우 필수 아님.

## 14.2 요청하지 않는 권한

- 현재 위치
- 항상 위치
- 백그라운드 위치
- 카메라
- 마이크
- 연락처
- 알림

알림은 연간 리캡 등 실제 필요가 생기는 후속 버전에서 선택적으로 검토한다.

## 14.3 제한 사진 접근

- 제한 접근에서도 사용 가능
- 자동 날짜 탐색 범위가 제한됨을 명확히 표시
- 시스템의 사진 선택 관리 UI로 이동하는 버튼 제공

## 14.4 개인정보 처리 원칙

- 사진 원본 업로드 없음
- 좌표 서버 전송 없음
- 사용자 계정 없음
- 외부 광고 식별자 없음
- 보드 데이터는 로컬 저장
- 외부 지도 타일 로딩 외 네트워크 의존 최소화

## 14.5 위치 노출 보호

공유 이미지에서 다음을 적용한다.

- 주거지 가능성이 높은 첫·마지막 장소 숨김 옵션
- 상세 주소 대신 장소명 또는 구 단위 표시
- 원본 EXIF는 렌더링 이미지에 포함하지 않음
- 공유 전 `세부 위치 숨기기` 토글 제공

# 23. 심사 및 운영 메모

## 심사 설명 핵심

- 앱은 사진의 촬영 위치와 시간을 사용해 정적 여행 보드를 생성함
- 현재 위치와 백그라운드 위치를 사용하지 않음
- 사진과 위치가 서버로 업로드되지 않음
- 무료 기능과 영구 구매 범위 명확히 표시

## 운영 부담

### 낮은 영역

- 서버 장애 없음
- 계정 문의 없음
- UGC 신고 없음
- 지도·장소 DB 자체 운영 없음

### 지속 관리 필요 영역

- iOS 사진 권한 변경 대응
- PhotoKit·MapKit 동작 변화
- 레이아웃 예외
- iCloud 원본 로딩
- 일본어 주소와 긴 지명
- 신규 기기 해상도

# 24. 제품 가드레일

다음 요구가 생겨도 초기 제품 방향을 유지한다.

1. 여행 계획 기능을 넣지 않는다.
2. 지도 검색·맛집 추천 앱으로 확장하지 않는다.
3. 공개 프로필과 팔로우를 넣지 않는다.
4. GPS 추적 정확도를 경쟁하지 않는다.
5. AI 채팅을 넣지 않는다.
6. 보드 편집기를 Canva 수준으로 만들지 않는다.
7. 구독을 기본 과금으로 전환하지 않는다.
8. 무료 사용자에게 광고 시청을 강제하지 않는다.

새 기능은 다음 질문을 통과해야 한다.

> 사용자가 여행 후 10초 안에 더 좋은 한 장을 만들게 하는가, 또는 다시 여행하고 기록하고 싶게 하는가?

둘 다 아니면 제외한다.

# 25. 최종 제품 문장

## 내부 정의

> MapRibbon은 사진 보관함 속 위치·시간 정보를 분석해, 하루의 여행을 지도 핀과 리본으로 연결한 공유용 포토보드로 자동 변환하는 로컬 iOS 앱이다.

## 한국 마케팅 문장

> **오늘 찍은 사진이, 한 장의 여행 지도가 됩니다.**

## 일본 마케팅 문장

> **今日の写真が、一枚の旅ボードになる。**

## 핵심 강점 5개

1. 여행 중 아무것도 켤 필요 없음
2. 날짜만 고르면 자동 완성
3. 실제 경로가 아닌 기억의 순서를 예쁘게 표현
4. 가입·광고·사진 업로드 없음
5. 사용할수록 한국·일본의 Memory Atlas가 채워짐
